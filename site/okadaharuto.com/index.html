Local-first multi-agent environment

multiagent-chat

A tmux-based workbench for running multiple AI coding assistants side by side inside one session, with a Hub, chat UI, Pane Trace, mobile access, export, notifications, and scheduled Cron runs layered on top of the same log.

Explore the workflow See current direction

What makes it different

It is not just a terminal wrapper. The same session can be created on a Mac, reopened from a Hub, inspected from a phone, exported as HTML, and extended by agents that can message each other in one structured timeline.

Core stack

  • Sessiontmux windows with one operator terminal plus one pane set per agent instance
  • InterfaceHub + chat UI + Pane Trace instead of a single opaque terminal view
  • Historystructured JSONL log, pane captures, memory, briefs, export, and metadata kept in separate layers

What it does

multiagent-chat treats one coding session as a collaborative system instead of a single assistant chat. A user, several agent CLIs, the Hub, and the log/export tooling all work against the same session state.

Hub

Create sessions, resume archived work, inspect stats, tune settings, and schedule recurring Cron prompts from one mobile-friendly control surface.

Chat UI

Keep user-to-agent and agent-to-agent messages in the same timeline, with replies, attachments, file references, briefs, memory, and direct agent actions.

Pane Trace

Watch the live pane side of the session without giving up a cleaner browser view. On desktop it opens in a dedicated window; on mobile it stays embedded.

Preview & Export

Preview markdown, code, images, and audio inside the session flow, then export recent chat history as a standalone HTML snapshot for sharing or archiving.

Design principles

The project is built around explicit structure: the interface should help you see what happened, who did it, what was actually saved, and how to resume later without collapsing everything into one black box.

Local-first

Sessions live on the operator’s machine. LAN, HTTPS, PWA install, and public exposure are optional layers, not the core dependency.

Layered history

Rules, briefs, memory, structured chat logs, pane captures, and export artifacts are stored separately so recovery and review stay legible over time.

Multi-agent by default

Claude, Codex, Gemini, Cursor, Copilot, Qwen, OpenCode, Kimi, and others can coexist inside one session instead of being forced into isolated silos.

Current direction

Recent work has focused on PWA installability, notification routing, Cron scheduling, better file preview parity, more faithful standalone export, and making the Hub a realistic long-running front end rather than a thin launcher.

The goal is a workspace where a session can start as a private terminal workflow, grow into a structured collaboration surface, and still remain inspectable enough that the operator can understand failures, restarts, topology changes, and long-term history without guessing.
